# Feature Specification: 共有URL読み込み修正

**Feature Branch**: `004-fix-shared-url`

**Created**: 2026-06-23

**Status**: Draft

**Input**: User description: "ループの共有でコピーしたURLでアクセスしても動作しないのを修正"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 共有URLで動画とループが正常に再生される (Priority: P1)

ユーザーAがLoopTubeでA-Bループを設定し「共有リンクをコピー」ボタンを押す。
ユーザーBがそのURLをブラウザで開くと、対象の動画が読み込まれ、タイムライン上のA/BマーカーとループのA点・B点・再生速度が正確に復元される。ユーザーBが再生を開始するとA-Bループが機能する。

**Why this priority**: これは共有機能の根幹であり、URLをコピーしても受信者がループを体験できないと機能として成立しない。

**Independent Test**: 共有URLをブラウザで開き、A/Bマーカーが正しい位置に表示され、再生速度が正しく設定されていることを確認することで完全にテスト可能。その後ユーザーが再生を開始するとA-Bループが正しく動作することで検証完了。

**Acceptance Scenarios**:

1. **Given** A-Bループが設定された状態で「共有リンクをコピー」を押してURLを取得, **When** そのURLを新しいブラウザタブで開く, **Then** 動画が読み込まれ、タイムライン上のA/BマーカーがコピーされたURL通りの位置に表示される。ユーザーが再生を開始するとA-Bループが機能する
2. **Given** 共有URLを開いた後, **When** ページが完全に読み込まれる, **Then** タイムラインのA/Bマーカーが正しい位置に表示されている
3. **Given** 再生速度が1.0以外（例: 1.5×）に設定されたループの共有URL, **When** そのURLを開く, **Then** 共有元と同じ再生速度がプレイヤーに設定される

---

### User Story 2 - 動画の長さより長い区間を含む共有URLのフォールバック (Priority: P2)

URLに含まれるA/B点が実際の動画の長さを超えている場合（例: 動画の長さ変更や不正なURL）でも、アプリがクラッシュせずに適切な区間でループが設定される。

**Why this priority**: エラー状態での適切な処理はUXの安定性に必要だが、正常ケースの修正が先決。

**Independent Test**: 動画の長さを超えるA/B値を含むURLでページを開き、クラッシュせずループが設定されることを確認できる。

**Acceptance Scenarios**:

1. **Given** 動画の長さ（例: 5分）を超えるB点（例: 10分）を含む共有URL, **When** そのURLを開く, **Then** B点が動画の終端にクランプされてループが設定され、クランプが発生したことをトースト通知（数秒で自動消滅）でユーザーに伝える

---

### Edge Cases

- 共有URLのパラメータが欠落・不正な場合はループが設定されず、通常の空白状態で起動する
- 動画IDのみ（`v`パラメータのみ）の場合は動画のみ読み込まれ、ループは設定されない
- YouTube動画が削除・非公開になった場合はプレイヤー側のエラーとして処理され、URL読み込み処理とは独立している
- 動画の読み込み完了前にループ区間のA/B点が設定されることはない（タイミング問題の防止）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは共有URLのクエリパラメータ（`v`, `a`, `b`, `s`）を解析し、ページ読み込み時に動画IDとループ設定を取得しなければならない
- **FR-002**: システムはYouTubeプレイヤーが動画を完全に読み込み、再生時間（duration）が確定した後にA/B点を設定しなければならない
- **FR-003**: A/B点の設定は動画の実際の長さの範囲内にクランプされた値で行われなければならない
- **FR-004**: 共有URLで動画が読み込まれた後、A点・B点・速度がプレイヤーに設定され、タイムラインのA/Bマーカーが正しい位置に表示されなければならない。再生の開始はユーザーの操作（タップ/クリック）に委ねる（モバイルブラウザの自動再生ブロックに対応するため）
- **FR-005**: 再生速度パラメータ（`s`）が有効な値の場合、共有元と同じ速度でループが再生されなければならない
- **FR-006**: URLに`v`パラメータのみ存在する場合、動画は読み込まれるがループは設定されないこと
- **FR-007**: URL解析に失敗した場合（必須パラメータ欠落・不正値）、アプリは通常の初期状態で起動し、エラー状態にならないこと

### Key Entities

- **共有URL**: `v`（動画ID）、`a`（A点秒数）、`b`（B点秒数）、`s`（再生速度）のクエリパラメータを持つURL
- **ループ設定**: A点・B点・再生速度の組み合わせ。動画の読み込み完了後に適用される
- **動画読み込みライフサイクル**: プレイヤー初期化 → 動画ロード → duration確定 → ループ設定の順序を保証する

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 有効な共有URLをブラウザで開いたとき、5秒以内にA/Bマーカーと再生速度が正しく復元され、タイムライン上に表示される
- **SC-002**: 共有URLで設定されたA点・B点の誤差が±0.1秒以内に収まる（正確な区間再現）
- **SC-003**: 共有URLを使用したユーザーの100%が、期待するループ区間の再生を体験できる（エラー・無応答なし）
- **SC-004**: 不正な共有URL（パラメータ欠落・不正値）を開いても、アプリがクラッシュせず通常の初期状態を表示する

## Clarifications

### Session 2026-06-23

- Q: 共有URL開封時の自動再生の挙動 → A: A/Bマーカーを正しい位置にセットし、再生はユーザーが手動で開始する（モバイルブラウザの自動再生ブロックに対応するため）
- Q: B点クランプ時のユーザー通知の形式 → A: トースト通知（既存の`shareToast`パターンを流用、数秒で自動消滅）

## Assumptions

- 動画プレイヤーの初期化後、動画の再生時間（duration）はプレイヤーが動画を実際にバッファリングし始めるまで利用できない。A/B点の設定はこのタイミング以降に行う必要がある。
- 共有URLのパラメータはLoopTubeの「共有リンクをコピー」機能が生成した形式であることを前提とし、外部からの不正なパラメータは適切に無視またはフォールバックする。
- ユーザーは共有URLを別ブラウザ・別端末で開くユースケースを想定する（認証状態・ループ保存履歴は引き継がれない）。
- 修正の対象は共有URLからのループ復元タイミングの問題であり、URL生成・解析ロジック自体の変更は最小限にとどめる。
