# Feature Specification: Bun化とRailwayデプロイ対応

**Feature Branch**: `002-bun-railway-deploy`

**Created**: 2026-06-22

**Status**: Draft

**Input**: User description: "リポジトリのbun化とRailwayのデプロイ対応"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 開発ツールのセットアップ高速化 (Priority: P1)

開発者がリポジトリをクローンした後、依存関係のインストールから開発サーバーの起動までを現在よりも大幅に速く完了できる。日常のワークフロー（インストール、ビルド、テスト実行）全体を通じて処理速度が向上する。

**Why this priority**: 開発者が最も頻繁に実行するタスクであり、ツールチェーンの高速化はチーム全体の生産性に直接かつ継続的に影響する。

**Independent Test**: 空のキャッシュ環境でリポジトリをクローンし、依存関係インストールから開発サーバー起動までの所要時間を計測することで独立してテスト可能。既存スクリプトも同様に動作確認できる。

**Acceptance Scenarios**:

1. **Given** 空のキャッシュ状態で, **When** 依存関係インストールコマンドを実行すると, **Then** インストールが正常に完了し、現在のpnpm環境と比較して30%以上速く終わる
2. **Given** 依存関係がインストール済みの状態で, **When** 開発サーバー起動コマンドを実行すると, **Then** 開発サーバーが起動し、LoopTubeのUIがブラウザで表示される
3. **Given** パッケージマネージャーが変更された状態で, **When** 既存スクリプト（`dev`, `build`, `test:unit`, `test:e2e`, `check`）を実行すると, **Then** すべてのスクリプトが変更前と同一の動作をする
4. **Given** CIパイプラインが実行される状態で, **When** テストスイートが走ると, **Then** 単体・統合・E2Eテスト全件が合格する

---

### User Story 2 - mainブランチへの自動デプロイ (Priority: P2)

運用担当者（または開発者）がmainブランチへコードをマージすると、アプリケーションが自動的にRailwayの本番環境にデプロイされ、公開URLで最新バージョンが利用可能になる。

**Why this priority**: 本番環境へのデリバリーパイプラインを確立することで、手動デプロイのリスクと工数を排除し、継続的デリバリーを実現する。

**Independent Test**: mainブランチへのプッシュ後、Railwayダッシュボードでビルドとデプロイのステータスおよびアプリの動作を確認することで独立してテスト可能。

**Acceptance Scenarios**:

1. **Given** mainブランチへ新しいコミットがプッシュされた状態で, **When** Railwayがビルドを実行すると, **Then** ビルドがエラーなく正常に完了する
2. **Given** ビルドが成功した状態で, **When** Railwayがデプロイを完了すると, **Then** 公開URLでアプリケーションが正常に表示される
3. **Given** アプリがRailwayにデプロイされた状態で, **When** ユーザーが公開URLを訪問すると, **Then** LoopTubeのUI（ヘッダー、URLフォーム、プレイヤー）が完全に動作する

---

### User Story 3 - 本番環境の設定管理 (Priority: P3)

運用担当者が本番環境に必要な設定値をRailwayのダッシュボード上で安全に管理でき、アプリケーションが本番設定を正しく読み込んで動作する。

**Why this priority**: アプリケーションの本番稼働に必要な環境固有の設定を、コードリポジトリに秘匿情報を含めずに管理するために必要。

**Independent Test**: Railway環境変数パネルで設定した値をアプリケーションが正しく読み込んでいるか、動作確認することで独立してテスト可能。

**Acceptance Scenarios**:

1. **Given** 必要な環境変数がRailwayで設定されている状態で, **When** アプリケーションが起動すると, **Then** 環境変数が正しく読み込まれ、アプリが期待通りに動作する
2. **Given** 設定が変更された状態で, **When** 再デプロイが実行されると, **Then** 新しい設定が反映された状態で動作する

---

### Edge Cases

- 既存のロックファイル（pnpm-lock.yaml）と新しいロックファイルが共存した場合の競合はどう扱うか？
- Railwayのビルドがタイムアウトした場合、開発者はどのように問題を検知・対処するか？
- 依存パッケージが新しい環境と非互換だった場合の挙動と回復手順は？
- デプロイ失敗時にロールバックできるか？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 開発者は1つのコマンドでプロジェクトの依存関係をインストールできなければならない
- **FR-002**: 開発者は1つのコマンドで開発サーバーを起動できなければならない
- **FR-003**: 既存のすべてのスクリプト（`dev`, `build`, `test:unit`, `test:e2e`, `check`等）が引き続き動作しなければならない
- **FR-004**: 既存の全自動テスト（単体・統合・E2E）が新しい環境で合格しなければならない
- **FR-005**: mainブランチへのプッシュがRailwayの自動デプロイをトリガーしなければならない
- **FR-006**: RailwayのデプロイURLでLoopTubeの主要機能（URL入力・動画読み込み・A-Bループ設定）が正常に動作しなければならない
- **FR-007**: 本番環境固有の設定（環境変数）がコードリポジトリに含めることなくRailway上で管理できなければならない
- **FR-008**: アプリケーションはRailway上でサーバープロセスとして動作しなければならない（将来のSSRおよびAPI routes対応を可能にするため）

### Key Entities

- **パッケージマネージャー設定**: プロジェクトルートの設定ファイル群とロックファイル
- **Railwayプロジェクト**: デプロイ先のクラウド環境（ビルド設定、環境変数、公開URL）
- **ビルド成果物**: 本番環境に配信されるコンパイル済みファイル群

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 依存関係のインストールが既存のpnpm環境と比較して30%以上速く完了する
- **SC-002**: mainブランチへのマージから30分以内にRailwayの公開URLでアプリケーションが利用可能になる
- **SC-003**: 既存の全自動テストスイート（単体・統合・E2E）が新しいツールチェーン環境で100%合格する
- **SC-004**: productionビルドがエラー0件で完了する
- **SC-005**: RailwayデプロイURL上でLoopTubeの主要機能（URL入力、動画読み込み、A-Bループ設定、セグメント保存）が正常に動作する

## Assumptions

- 現行のSvelteKitプロジェクト構成（ルーティング・コンポーネント構造）は変更しない
- GitHubリポジトリとRailwayを連携して自動デプロイを設定する
- Railwayの標準プランで十分な性能要件を満たせる
- 本番環境で必要な外部APIキーは現時点では存在しない（YouTube IFrame APIはクライアントサイドのみで動作する）
- **Constitution改訂が前提条件**: 現行のConstitutionはパッケージマネージャーを「npm または pnpm」と定めており、bunへの移行にはConstitutionの改訂が必要である。本フィーチャーの実施前にConstitutionを改訂するか、または本フィーチャーの一部としてConstitution改訂を含めることを前提とする
